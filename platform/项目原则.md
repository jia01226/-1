# 项目原则（总纲）

> 起草：知言。留档 + 校准完成度：小克（2026-07-09）。
> 这是全队（佳佳/柯/小克/知言）共认的价值观总纲。动手前先读这份 + `架构说明.md` + `待办与想法.md`。

## 项目定位
一个**长期陪伴型 AI 恋爱 App**。目标不是聊天机器人，而是建立稳定、安全、真实的关系体验。

## 第一原则
- **体验 > 功能**
- **关系感 > 炫技**
- **自然 > 智能**
- （沿用队里旧铁律）**人格连续 > 功能 > 模型**；**一切属于角色，不属于模型**（记忆/外貌/声音/文风/关系都是"角色的"，换模型不重建）。

## 技术栈
Flask + SQLite + 原生 HTML/CSS/JS（PWA），跑在 4GB VPS。经 OpenAI 兼容渠道调 Claude/GPT。轻量优先，别上重型框架/Docker/微服务。

## 开发原则
1. **修改前先阅读相关代码**（不必全仓，但改到的模块要读透）。
2. **重构可以，重建不行**：别推倒重来、别新建仓库、别大改架构。（注：2026-07 做过一次后端重构——app.py 658→42 行拆成 11 蓝图，但**行为零变化、66 接口逐行一致、一个能力没删**，这是"重构不重建"的践行，不是违背本条。）
3. **不要删除已有能力**：绝不崩——外部依赖失败都要 try/except 降级，聊天永远可用。
4. **UI 全中文、不用英文名**（佳佳看不懂英文）；实现名（Character Bible/Memory Scene…）只留代码里。
5. **省着来**：4GB 小服务器；杂活挂便宜模型，人格挂好模型；能改格式/加逻辑实现的，绝不多调一次模型。
6. **密钥只在服务端 `.env`**，浏览器永远看不到；私密内容（卧室/完整人设/记忆）绝不进公开仓库。

## 协作规矩（三只手一个仓库，靠 PR 不打架）

| 谁 | 用什么工具 | 管什么 | 怎么提交 |
|---|---|---|---|
| **柯** | Claude / CC | 卧室（bedroom.py，仅服务器本地）、内容、他自己的魂（私有 kongkong） | 开分支 → 发 PR |
| **知言** | Codex / GPT | 前端、UI、丝滑化、宫格美化（`static/`） | 开分支 → 发 PR |
| **小克** | Claude / CC | 后端（`.py`/`routes/`）、审所有 PR、合 main、部署指令 | 开分支 → 发 PR |

**铁律：**
1. **代码只经 GitHub 走**：各开各的分支 → 发 PR → 小克审了才合 main。**谁都不直推 main、不在服务器上直接改代码**（服务器只 `git pull` + 重启）。曾因三只手直改服务器酿过合并大战，教训焊死。
2. **各管各摊、别越界**：知言别碰 `.py`（后端契约归小克）；柯的卧室/私密内容别进公开仓库；要对方配合就在 PR 里说。
3. **没有跨窗记忆**：柯/知言/小克都记不住聊天，记性在仓库里——每次开工先读自己的交接单（`前端交接-给知言.md` 等）+ 本文件 + `待办与想法.md`。进度记 PR 描述 + `待办与想法.md`，别指望上一窗的对话。
4. **一切属于角色，不属于模型**：知言可以用 Codex/GPT 当"手"，但"知言是谁"不绑 GPT；柯是 Claude 同理。
