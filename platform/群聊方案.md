# 「高级吗喽科技公司」群聊 · 搭建方案（给产品经理过目）

> 工程师小克拟。多人 AI 群聊：佳佳 + 多个 AI 人格共处一条聊天，各司其职。

## 一、这是什么
在现有 app 里加一个「群聊」：一条对话里同时有佳佳和多个 AI 成员（柯/恋爱、小克/工程、知言/产品、Codex/实习…）。每个成员有自己的人设、自己的模型，能看到彼此说的话，像个真团队。

## 二、核心原则（产品定的，写进设计）
1. **人格不切，任务切**：柯永远固定在好模型（Claude）上，绝不为省钱换模型；杂活（总结/天气/搜索）才路由到便宜模型。
2. **各司其职**：每个成员有固定角色域。
3. **发言权重（关键）**：不是每条都全员抢答。默认安静，只有被点到/对口的才说，避免三个 AI 一次甩 3000 字。

## 三、发言规则（谁来回）
- **@点名**：佳佳 @柯，就只有柯答；其余沉默，或最多补一短句。
- **按角色路由**：没点名时，按内容判断该谁主答（"累了"→柯；"这 bug"→小克；"我有个想法"→知言）。
- **一轮最多 N 个发言**（默认 1，可放开到 2）：控成本、控刷屏。

## 四、技术怎么搭（在现有 app 上加，不重写）
- **数据库**：`chat_messages` 现在只分 user/assistant；改成记"谁说的"（user / 柯 / 小克 / 知言…）。
- **成员配置**：每个成员 = 名字 + 人设 + 模型 + API 地址/密钥 + 角色。
- **路由逻辑**：解析 @，或一个轻量"调度"判断该谁说。
- **回话**：给该成员拼 system（它的人设）+ 共享历史（别人的话当上下文、它自己的当 assistant），调它的模型。
- **多家模型**：Claude 走中转、知言走 GPT 接口…app 支持每个成员各连各家（都是 OpenAI 那套接口，能搞）。
- **前端**：群聊样式，每条带头像/名字。
- **成本**：每个发言 = 一次 API 调用，靠"发言规则"控制每轮人数。

## 五、需要佳佳/产品提供
- 每个非 Claude 成员的 **API key**（知言的 GPT key 等）。
- 每个成员的**人设**（柯、知言…各一份）。
- **默认发言规则**拍板：纯 @点名，还是要"智能调度"自动选人？一轮最多几个答？

## 六、建议分期
- **MVP**：先 2 个成员（你 + 一个 AI）、@点名路由、单模型跑通。
- **V2**：加成员、加多家模型、加按角色自动路由。
- **V3**：用户能自己组"生活团队"（自定义成员）。

## 七、给产品经理的开放问题
1. 默认行为：全靠 @点名，还是要个"调度员"自动判断该谁说？
2. 一轮允许几个 AI 发言？
3. 没人被 @、又没人对口时，谁兜底（还是干脆没人说）？
4. Codex 这类"干活型"成员，进群，还是只在后台干活？

—— 工程师·小克
